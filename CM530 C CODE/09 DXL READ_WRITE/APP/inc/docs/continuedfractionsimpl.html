Numbers as streams


One approach to exact real arithmetic is using streams as a data structure to represent numbers. Even if a machine has finite resources and cannot store an infinite sequences, such infinite streams can represent programs that successively generate the elements of the sequence. The stream computed on demand are often referred to as computable streams. Some possible representations are continued fractions, B-adic number streams, linear fractional transformations and continued logarithms (a mutation of continued fractions that instead of the integer partial quotients uses only 0's and 1's).

A continued fraction is actually a stream of numbers. The arithmetic with continued fractions is harder than the arithmetic on numbers in positional notations, but some advantages arise immediately. First of all, some irrational numbers can be represented exactly as little programs to provide the partial quotients on demand. Another advantage is the possibility to cascade several routines and produce an output stream of partial quotients at a low cost. The user doesn't have to decide in advance how much precision it necessary, preventing wasting time and effort on more precision than it is actually required.

Currently, we support only simple continued fractions, (simple because all the numerators are one) but expecting to add nonregular continued fraction support in future versions of the project. We do not actually use java.io.Stream to represent a continued fraction, as one might expect. Instead, we derive our numbers from java.util.Iterator, usually iterating over the partial quotients of continued fraction expansions.

The ContinuedFraction represents a simple continued fraction that can be reused and supports the elementary arithmetic operations. The abstract class PartialQuotients interface represents a (potentially infinite) stream of partial quotients. While a PartialQuotients instance is consumable, a ContinuedFraction knows to generate its partial quotients as many times as required. ContinuedFraction will probably be refactored to implement Number in future versions.
